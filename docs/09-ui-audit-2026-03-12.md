# ToneMatch UI Audit

Date: March 12, 2026
Scope: `apps/mobile` across Expo native patterns and the supported web surface (`start_expo.sh web`)
Method: source review, `npm run typecheck:mobile`, `npm run lint:mobile`, live inspection in Expo web at `390x844` and `1280x900`

## Anti-Patterns Verdict

Verdict: Fail

This does not look like generic 2024-2025 AI slop at first glance. The warm palette and Cormorant/Manrope pairing are more intentional than the usual purple-gradient template. The problem is structure. The app repeats the same safe pattern across Home, Discover, and Paywall:

- oversized editorial hero image
- uppercase overline
- stacked cards underneath
- repeated product or feature tiles

Specific tells from the frontend-design anti-pattern list:

- too many card containers driving hierarchy instead of layout
- repeated hero-card-plus-card-stack composition
- repeated editorial copy that restates visible context instead of sharpening it
- web layouts that mostly stretch mobile compositions instead of adapting to larger contexts

The visuals are better than average AI output, but the composition system still feels AI-assisted and template-led.

## Executive Summary

- Total issues: 12
- Critical: 1
- High: 5
- Medium: 4
- Low: 2
- Overall quality score: 5.8/10

Top issues:

1. Informative imagery is effectively silent to assistive tech across the app.
2. Dark mode support is architecturally present but functionally bypassed in most screens.
3. Core web surfaces regress badly: Home CTA cards collapse and Paywall content overlaps.
4. Automated guardrails are broken: both typecheck and lint fail.
5. The web layout does not meaningfully adapt beyond phone width.

Recommended next steps:

1. Use `imp_harden` on accessibility and web interaction defects first.
2. Use `imp_adapt` to fix web/mobile responsive behavior.
3. Use `imp_normalize` to migrate all screens to theme-aware tokens.
4. Use `imp_optimize` after layout fixes to reduce image weight and animation overhead.

## Detailed Findings By Severity

### Critical Issues

#### 1. Informative images have no accessible alternative text

- Location: [apps/mobile/app/(tabs)/home.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/(tabs)/home.tsx#L131), [apps/mobile/app/paywall.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/paywall.tsx#L172), [apps/mobile/app/(tabs)/discover.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/(tabs)/discover.tsx#L771), [apps/mobile/app/style-guide.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/style-guide.tsx#L128)
- Severity: Critical
- Category: Accessibility
- Description: The app contains 20 `Image` usages across 11 files, but the codebase only exposes 2 image-related accessibility labels, and those belong to capture actions rather than the images themselves.
- Impact: Screen reader users miss hero imagery, product imagery, guide imagery, and scan previews that carry context or meaning. On web this also leaves the accessibility tree largely devoid of non-text content descriptions.
- WCAG/Standard: WCAG 2.2 1.1.1 Non-text Content (Level A)
- Recommendation: Mark decorative images as hidden from accessibility and add meaningful labels for informative images. Product and guide imagery should describe what the image conveys, not restate the filename.
- Suggested command: `imp_harden`

### High-Severity Issues

#### 2. Dark mode is largely broken because most screens bypass the active theme

- Location: [apps/mobile/src/theme/theme-provider.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/src/theme/theme-provider.tsx#L17), [apps/mobile/app/(tabs)/_layout.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/(tabs)/_layout.tsx#L6), [apps/mobile/app/(tabs)/home.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/(tabs)/home.tsx#L21), [apps/mobile/app/paywall.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/paywall.tsx#L21)
- Severity: High
- Category: Theming
- Description: `ThemeProvider` correctly exposes light and dark palettes, but 22 files still import the static light `palette` constant directly. Those screens will not react to theme changes.
- Impact: Users switching to dark mode get a mixed interface where shared themed components adapt but route-level screens stay in light colors. This produces inconsistent contrast, visual seams, and a broken theme contract.
- WCAG/Standard: Design system consistency / platform theming expectations
- Recommendation: Remove direct `palette` imports from route files and derive route styles from `useAppTheme` or `useThemedStyles`.
- Suggested command: `imp_normalize`

#### 3. Home screen quick-action cards collapse into tiny text links on web

- Location: [apps/mobile/app/(tabs)/home.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/(tabs)/home.tsx#L199), [apps/mobile/app/(tabs)/home.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/(tabs)/home.tsx#L488)
- Severity: High
- Category: Responsive
- Description: In live web inspection at `390x844`, the `Link asChild` quick actions render as narrow anchors instead of full cards. The “New Scan”, “Quick Check”, and “Guides” actions measure roughly text-width by `43px` height rather than occupying the intended card area.
- Impact: Three primary navigation actions lose visual affordance and one of them drops below the 44px minimum touch target height. This makes core flows harder to find and tap on web.
- WCAG/Standard: WCAG 2.2 2.5.8 Target Size (Minimum), usability regression on supported web surface
- Recommendation: Verify the `Link` + `Pressable` composition on web and ensure the clickable node receives the card dimensions, background, and layout styles.
- Suggested command: `imp_adapt`

#### 4. Paywall content overlaps on mobile web, making upgrade messaging unreadable

- Location: [apps/mobile/app/paywall.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/paywall.tsx#L170), [apps/mobile/app/paywall.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/paywall.tsx#L188), [apps/mobile/app/paywall.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/paywall.tsx#L374), [apps/mobile/app/paywall.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/paywall.tsx#L402)
- Severity: High

- Impact: The paywall’s benefits section becomes harder to read at exactly the viewport most likely to represent phone usage on web. That directly damages the subscription funnel.

#### 6. The web surface lacks meaningful semantic structure and landmarks

- Location: [apps/mobile/src/components/screen.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/src/components/screen.tsx#L22), [apps/mobile/app/(tabs)/home.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/(tabs)/home.tsx#L95), [apps/mobile/app/(tabs)/discover.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/(tabs)/discover.tsx#L600)

- Category: Accessibility
- Description: Live accessibility snapshots for Home, Discover, and Paywall are almost entirely `generic` containers with little heading or landmark structure.

### Medium-Severity Issues

#### 7. Several text token combinations miss AA contrast

- Location: [apps/mobile/src/theme/palette.ts](/home/aytzey/Desktop/ToneMatch/apps/mobile/src/theme/palette.ts#L15), [apps/mobile/app/analysis-loading.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/analysis-loading.tsx#L767), [apps/mobile/app/analysis-loading.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/app/analysis-loading.tsx#L781)
- Severity: Medium
- Category: Accessibility
- Description: The palette contains text colors that miss 4.5:1 in normal text usage. Verified examples:

- Impact: Small captions, pending states, and secondary labels become harder to read, especially for low-vision users and in bright environments.
- WCAG/Standard: WCAG 2.2 1.4.3 Contrast (Minimum)

## Patterns And Systemic Issues

- Static light-theme palette usage appears in 22 files, so the theming problem is systemic.
- The app uses many thoughtful accessibility labels on buttons, but accessible treatment for visual content is largely missing.
- Web support exists operationally, but several route compositions are still effectively native-first and do not survive React Native Web translation cleanly.
- Large editorial assets are used as primary differentiation across many screens, which increases both payload size and template repetition.

## Positive Findings

- Typography is stronger than average. Cormorant + Manrope gives the product a specific editorial tone instead of defaulting to Inter/system.
- The palette direction is cohesive and avoids the obvious AI-gradient/purple-neon trap.
- Most interactive controls are intentionally sized at or near `44x44`, and many `Pressable` elements include explicit accessibility labels and states.
- Motion is generally using transform/opacity patterns with reduced-motion support, which is the right baseline architecture.
- Shared primitives such as [apps/mobile/src/components/primary-button.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/src/components/primary-button.tsx#L35) and [apps/mobile/src/components/editorial-guide-primitives.tsx](/home/aytzey/Desktop/ToneMatch/apps/mobile/src/components/editorial-guide-primitives.tsx#L23) show stronger implementation discipline than many route files.

## Recommendations By Priority

1. Immediate
- Fix missing image accessibility treatment.
- Repair Home web CTA rendering and Paywall overlap.
- Restore passing lint and typecheck.

2. Short-term
- Migrate route files off the static `palette` import.
- Add web semantics for headings, sections, and navigation.
- Audit and tighten low-contrast token usage.

3. Medium-term
- Add actual tablet/desktop compositions for Home and Discover.
- Reduce asset weights and replace oversized PNGs.
- Normalize web animation and deprecated prop usage.

4. Long-term
- Reduce structural repetition in screen composition so the editorial direction comes from layout decisions, not only from imagery and copy tone.

## Suggested Commands For Fixes

- `imp_harden`
  - best first follow-up for image accessibility, semantic structure, validation states, and broken guardrails
- `imp_adapt`
  - best for Home web CTA collapse, Paywall overlap, and desktop layout re-composition
- `imp_normalize`
  - best for dark-mode migration and token cleanup
- `imp_optimize`
  - best for image compression, web animation cleanup, and rendering efficiency
- `imp_clarify`
  - best for auth and paywall copy/error-state cleanup
- `imp_polish`
  - final pass after structural fixes to tighten spacing, visual rhythm, and hierarchy
